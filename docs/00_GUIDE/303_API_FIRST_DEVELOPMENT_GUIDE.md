# API 설계 우선 개발 가이드라인 (API-First Development Guide)

> **프로젝트:** ZENITH_LMS (SNTL 통합 물류 플랫폼)
> **문서번호:** Gd-303
> **작성자:** Antigravity (AI Agent)
> **작성일:** 2026-04-22
> **버전:** v1.0

본 문서는 ZENITH_LMS 프로젝트에서 API 설계 우선(API-First) 원칙에 따라 기능을 개발하는 구체적인 절차와 표준을 정의합니다.

## 1. 개요 (Background)
물류 시스템과 같은 복잡한 비즈니스 로직을 다루는 프로젝트에서 코드 구현을 먼저 진행할 경우, 프론트엔드와 백엔드 간의 인터페이스 불일치, 중복 로직 발생, 테스트 자동화의 어려움이 발생합니다. 이를 방지하기 위해 **모든 기능은 승인된 API 명세를 기반으로 구현**되어야 합니다.

## 2. API 개발 워크플로우 (Workflow)

### 단계 1: 요구사항 분석 및 스키마 설계
- 새로운 기능 요청 시, 관련 데이터 모델(Table)과 필드 사양을 먼저 정의합니다.
- Zod를 사용한 입력 데이터 검증 스키마를 설계합니다.

### 단계 2: API 명세서(Ds-11) 작성 및 업데이트
- `docs/03_Design/Ds_11_API_상세_명세서.md`에 해당 API를 등록합니다.
- **필수 포함 항목**:
    - **ID/명칭**: Action 또는 RPC 함수의 이름
    - **방식**: `Action` (Next.js Server Action) 또는 `RPC` (Supabase Postgres Function)
    - **권한**: Public, User, Admin 등 접근 제한 설정
    - **입력 데이터**: 파라미터명, 타입, 필수 여부, 설명 (Zod 스키마 연계)
    - **반환 데이터**: 성공 시 반환되는 객체 구조 (JSON 형식 권장)
    - **에이전트**: 해당 API를 호출/처리하는 비즈니스 로직 설명

### 단계 3: 사용자 승인
- 구현 전, 작성된 명세서를 사용자(Master Edward)에게 보고하고 설계 승인을 득합니다.

### 단계 4: 테스트 케이스 작성 (TDD)
- 승인된 명세를 기반으로 실패하는 테스트 코드를 먼저 작성합니다.

### 단계 5: 구현 (Implementation)
- 명세에 정의된 인터페이스를 정확히 준수하여 코드를 작성합니다.
- `src/app/actions/` 하위의 관련 도메인 파일에 구현합니다.

### 단계 6: 자가 검증 및 동기화 확인
- 구현 완료 후, 실제 반환값과 입력값이 명세서와 일치하는지 확인합니다.

## 3. 에이전트 준수 사항 (Compliance)
- **독단적 구현 금지**: 명세서에 없는 필드를 추가하거나 타입을 임의로 변경하여 구현하지 않습니다.
- **문서 선행 업데이트**: 로직 변경으로 인해 인터페이스가 변해야 하는 경우, 코드를 수정하기 전 명세서부터 수정하고 보고합니다.
- **에러 처리 표준화**: 모든 API는 `Ds_11` 섹션 1.1에 정의된 공통 응답 구조를 준수해야 합니다.

## 4. 인벤토리 관리 특화 규칙
- 오더와 관련된 모든 API는 `syncInventoryFromOrder`와 같은 재고 동기화 함수를 적절한 시점에 트리거해야 합니다.
- 재고 수량 부족 등 비즈니스 예외 발생 시, 사용자에게 명확한 에러 메시지를 반환해야 합니다.
