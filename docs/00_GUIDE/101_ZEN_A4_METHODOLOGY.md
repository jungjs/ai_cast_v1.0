# ZEN_A4 (A++++) - 자동화 개발 방법론

> 자동화된 코드 리뷰, 조건부 GSD 트리거, 품질 게이트 시스템  
> **작성일:** 2026-04-08  
> **버전:** 1.0

---

## 📌 ZEN_A4란?

**ZEN_A4** = **A++++** (기존 약자를 명명)

3가지 자동화 훅으로 구성된 **코드 품질 자동화 시스템**

```
목표: 개발자가 코드 작성만 집중하고, 
      리뷰/검증/품질 관리는 자동화
```

---

## 🎯 핵심 역할 (3가지 훅)

### 1️⃣ PreToolUse (코드 작성 전)

**목적**: 복잡도가 높은 기능이나 보안 코드 감지 → 자동으로 적절한 에이전트 실행

| 감지 유형 | 자동 실행 | 효과 |
|----------|---------|------|
| **복잡한 기능** | `/gsd-plan-phase` | 설계 단계 자동 수행 |
| **보안 코드** | `security-reviewer` | 보안 검증 자동 수행 |
| **아키텍처 변경** | `architect` | 아키텍처 리뷰 자동 수행 |
| **대규모 리팩토링** | `refactor-cleaner` | 죽은 코드 자동 정리 |

**효과**: 개발자가 놓칠 수 있는 중요한 검증 자동 수행

---

### 2️⃣ PostToolUse (코드 작성 후)

**목적**: 모든 코드 작성 후 자동으로 품질 검증 수행

| 코드 유형 | 자동 실행 | 검증 항목 |
|----------|---------|---------|
| **모든 코드** | `code-reviewer` | 가독성, 패턴, 최적화 |
| **보안 관련 코드** | `security-reviewer` | 취약점, 인젝션, 암호화 |
| **테스트 코드** | `tdd-guide` | 커버리지, 격리, 모킹 |
| **문서** | `doc-updater` | 완성도, 가독성, 구조 |
| **Language-Specific** | 언어별 리뷰어 | 언어 관례, 성능, 안티패턴 |

**효과**: 사람 리뷰어처럼 자동으로 코드 품질 검증

---

### 3️⃣ Stop (커밋 전)

**목적**: 최종 품질 게이트 - 모든 기준을 만족하지 않으면 커밋 차단

| 검증 항목 | 기준 | 상태 |
|----------|------|------|
| ✅ 테스트 커버리지 | 80% 이상 | 필수 |
| ✅ Linting | 통과 | 필수 |
| ✅ 보안 스캔 | 위반 없음 | 필수 |
| ✅ Self Check 통과 | 설계 검증 완료 | 필수 |
| ✅ Self Test 통과 | 기능 테스트 완료 | 필수 |
| ✅ SAR 작성 | 오류 발견 시 필수 | 조건부 |

**효과**: 불완전한 코드의 커밋 원천 차단

## 🔄 ZEN_A4 + GSD 통합 워크플로우

```
┌─────────────────────────────────────────┐
│ 기능 개발 요청                            │
└──────────────┬──────────────────────────┘
               ↓
        복잡도 판단?
        ├─ 복잡함 → PreToolUse 감지
        │          ↓
        │     /gsd-plan-phase 자동 실행
        │          ↓
        │     [설계 문서 생성]
        │
        └─ 간단함 → 직접 코딩
               ↓
        ┌──────────────────────┐
        │ 코드 작성             │
        └──────────────┬───────┘
                       ↓
        ┌──────────────────────────────┐
        │ PostToolUse (자동 리뷰)      │
        │ - code-reviewer              │
        │ - security-reviewer          │
        │ - language-reviewer          │
        └──────────────┬───────────────┘
                       ↓
        ┌──────────────────────────────┐
        │ Self Check & Self Test       │
        │ (개발자 수동 검증)            │
        │ - 설계 검증                  │
        │ - 기능 테스트                │
        │ - 커버리지 확인              │
        └──────────────┬───────────────┘
                       ↓
        ┌──────────────────────────────┐
        │ Phase 3: Audit (Verify)      │
        │ (Audit Agent - 정밀 감사)     │
        │ - Gemini 3.1 Pro (High)      │
        │ - SAR 및 정합성 최종 감사    │
        └──────────────┬───────────────┘
                       ↓
             오류 발견?
             ├─ YES → SAR 작성
             │        Check List 업데이트
             │        코드 수정 & 재테스트
             │
             └─ NO
                  ↓
        ┌──────────────────────────────┐
        │ Stop (최종 품질 게이트)       │
        │ - 커버리지 80%+              │
        │ - Linting 통과               │
        │ - 보안 검증 통과             │
        │ - SAR(필요시) 완료           │
        └──────────────┬───────────────┘
                       ↓
             모든 기준 만족?
             ├─ YES → 커밋 허용 ✅
             └─ NO  → 개발자에게 반려 ❌
```

## 📊 ZEN_A4의 효과

### 자동화율

| 항목 | 자동화율 | 절약 시간 |
|------|---------|---------|
| 코드 리뷰 | 80%+ | 함수당 3-5분 |
| 설계 검증 | 조건부 (복잡시만) | 기능당 10-30분 |
| 보안 검증 | 100% | 함수당 2-3분 |
| 문서 관리 | 80%+ | 파일당 5분 |
| **전체** | **80%+** | **함수당 10-15분** |

### 품질 향상

| 지표 | 개선 효과 |
|-----|---------|
| 버그 감소 | -30~40% |
| 코드 리뷰 반려율 | -50% |
| 테스트 커버리지 | 80%+ 자동 유지 |
| 보안 이슈 | -60~70% |
| 개발 신뢰도 | +40% |

## 🛠️ ZEN_A4 설정

### settings.json 예시

```json
{
  "hooks": {
    "preToolUse": {
      "enabled": true,
      "rules": [
        {
          "trigger": "complex_feature_detected",
          "action": "/gsd-plan-phase",
          "description": "복잡한 기능 → 자동 설계"
        },
        {
          "trigger": "security_code_detected",
          "action": "security-reviewer",
          "description": "보안 코드 → 자동 검증"
        }
      ]
    },
    "postToolUse": {
      "enabled": true,
      "autoReview": true,
      "agents": [
        "code-reviewer",
        "security-reviewer",
        "language-specific-reviewer"
      ]
    },
    "stop": {
      "enabled": true,
      "qualityGate": {
        "testCoverage": "80%",
        "linting": "required",
        "security": "required",
        "selfCheck": "required",
        "selfTest": "required"
      }
    }
  }
}
```

## ✅ 운영 가이드

### 1. PreToolUse 활용

**복잡한 기능 판단 기준**:
- 새로운 아키텍처 도입
- 5개 이상의 파일 수정
- 외부 API 통합
- 상태 관리 전반적 변경

**동작**:
```bash
# 개발자는 코딩만 시작
# → 복잡도 감지되면 자동으로 /gsd-plan-phase 실행
# → 설계 문서 자동 생성
```

### 2. PostToolUse 활용

**자동으로 수행되는 검증**:
```bash
1. 코드 작성 완료
   ↓
2. code-reviewer 자동 실행
3. security-reviewer 자동 실행
4. language-reviewer 자동 실행
   ↓
5. 리뷰 결과 자동 제시
```

### 3. Stop 활용

**커밋 전 최종 확인**:
```bash
git commit -m "feat: 새 기능"
   ↓
Stop 훅 실행
   ├─ 커버리지 확인 (80%+?)
   ├─ Linting 확인 (통과?)
   ├─ 보안 스캔 확인 (안전?)
   └─ Self Check/Test 완료 (통과?)
   ↓
모두 만족 → 커밋 허용 ✅
하나라도 미충족 → 커밋 거부 ❌
```

## 📌 Best Practices

### DO ✅

- ✅ Self Check/Test를 먼저 통과시키고 커밋
- ✅ 자동 리뷰 결과를 신뢰하고 개선
- ✅ 복잡한 기능은 설계 단계부터 시작
- ✅ SAR을 작성하고 Check List 업데이트
- ✅ PreToolUse 규칙을 정기적으로 검토

### DON'T ❌

- ❌ Stop 게이트를 무시하고 강제 커밋
- ❌ 자동 리뷰 없이 코드 작성 완료 선언
- ❌ SAR 작성을 생략
- ❌ 테스트 커버리지 80% 미달로 커밋
- ❌ 보안 검증 미통과 상태로 진행
